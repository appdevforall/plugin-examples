1. Executive Overview

Beepy is a lightweight CodeOnTheGo plugin that delivers audible feedback during builds. It plays a distinct short sound (earcon) when a build starts, succeeds, or fails, letting developers keep their eyes on code while staying aware of build progress.

The plugin is implemented in a single Kotlin file (~80 lines), requires no special permissions, and ships three bundled WAV files as its only assets. It integrates with the IDE through two framework interfaces — IPlugin for lifecycle management and BuildStatusListener for build events — making it a clean, minimal example of event-driven plugin design.

2. Core Functionality

Build-event sound mapping

Build EventCallbackSound File
Build startsonBuildStarted()res/raw/started.wav
Build succeedsonBuildFinished()res/raw/finished.wav
Build failsonBuildFailed(error)res/raw/failed.wav

Audio behaviour

  • Sounds are routed through USAGE_ASSISTANCE_SONIFICATION, so they follow media/ringer volume and respect Do Not Disturb rules rather than interrupting like a notification.
  • The SoundPool is configured with a maximum of two concurrent streams, allowing overlapping playback if events arrive in quick succession.
  • Playback parameters: full stereo volume (1.0 L / 1.0 R), priority 1, no looping, normal playback rate.

Graceful degradation

SoundPool.load() is asynchronous. If a build event fires before decoding finishes (~few hundred ms after activation), the event is silently skipped. Because builds almost always outlast the load time, this is rarely observable in practice.

Key implementation details

Sound loading

private fun loadSound(key: String, @RawRes rawId: Int) {
    val pool = soundPool ?: return
    val id = context.androidContext.resources.openRawResourceFd(rawId).use {
        pool.load(it, 1)
    }
    soundIds[key] = id
}

Each WAV is opened as a raw resource file descriptor and handed to SoundPool.load(). The returned sample ID is stored in soundIds keyed by event name. When decoding completes, the OnLoadCompleteListener adds the sample ID to loadedIds.

Guarded playback

private fun play(key: String) {
    val pool = soundPool ?: return
    val id = soundIds[key] ?: return
    if (id !in loadedIds) return
    pool.play(id, 1f, 1f, 1, 0, 1f)
}

Three guards protect against premature or invalid playback: the pool must exist, the sound ID must be mapped, and the sample must have finished loading. Only then is play() called.

Resource cleanup

override fun deactivate(): Boolean {
    context.services.get(IdeBuildService::class.java)
        ?.removeBuildStatusListener(this)
    soundPool?.release()
    soundPool = null
    soundIds.clear()
    loadedIds.clear()
    return true
}

On deactivation the listener is removed first, then the native SoundPool is released and all tracking state is cleared. dispose() is a no-op since deactivate() already handles full cleanup.

4. Integration Points

Beepy touches the host IDE through a small number of well-defined interfaces. It does not modify any main-app source files — all integration happens at runtime through the plugin framework.

4.1 Plugin framework (IPlugin)

The Beepy class implements com.itsaky.androidide.plugins.IPlugin, which the IDE discovers and manages. The manifest declares the entry point: 

<meta-data android:name="plugin.main_class"
           android:value="com.example.beepy.Beepy" />

The IDE calls initialize() → activate() on load, and deactivate() → dispose() on removal, giving the plugin full lifecycle control.

4.2 Build service (IdeBuildService / BuildStatusListener)

On activation, the plugin retrieves IdeBuildService from the host's service registry and registers itself as a BuildStatusListener: 

context.services.get(IdeBuildService::class.java)
    ?.addBuildStatusListener(this)

This is the only service the plugin consumes. The three callback methods — onBuildStarted, onBuildFinished, and onBuildFailed — each map directly to a sound.

4.3 Android context (resource access)

The plugin accesses context.androidContext.resources to open its bundled WAV files via openRawResourceFd(). This is the only use of the Android context; the plugin creates no Activities, Services, BroadcastReceivers, or ContentProviders.

4.4 Permissions

Beepy requires no plugin permissions. The available permission system (filesystem.read, filesystem.write, network.access, system.commands, ide.settings, project.structure) is left entirely unused — the plugin only reads its own bundled resources and listens to build events.

5. Deployment & Usage

Building

./gradlew :Beepy:assembleRelease

This produces a .cgp plugin package under Beepy/build/outputs/.

Installation

  1. Open CodeOnTheGo's plugin manager.
  2. Import the .cgp file.
  3. The IDE discovers Beepy via the manifest metadata and activates it automatically.

Customising sounds

  1. Replace the WAV files in src/main/res/raw/ using the same filenames: started.wav, finished.wav, failed.wav.
  2. Keep clips short — SoundPool is designed for samples under ~1 MB decoded.
  3. Rebuild and reinstall the plugin.

6. Key Benefits

  • Eyes-free build awareness. Developers working on a phone or tablet can stay focused on the editor and still know instantly when a build starts, finishes, or fails.
  • Minimal footprint. One Kotlin file, three small WAVs, zero permissions. The plugin adds negligible size and no runtime overhead outside of the brief moments when a sound plays.
  • Respectful audio routing. By using USAGE_ASSISTANCE_SONIFICATION, sounds follow the device's media volume and DND rules instead of interrupting like notifications.
  • Clean lifecycle management. Resources are allocated lazily on activation and fully released on deactivation — no leaks, no lingering listeners.
  • Easily customisable. Swapping the three WAV files is the only step needed to personalise the audio cues.
  • Good reference implementation. The plugin demonstrates the core patterns needed to write a CodeOnTheGo plugin — implementing IPlugin, accessing the service registry, subscribing to events, and managing plugin lifecycle — in under 80 lines of code.
