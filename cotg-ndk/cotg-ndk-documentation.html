1. Executive Overview

cotg ndk is a native C++ build environment that runs on the device. It installs an Android NDK, discovers the Clang toolchain and sysroot, provisions missing Android platform libraries, configures the active project's app/src/main/cpp with CMake, compiles the native sources for arm64-v8a, and writes the resulting shared library into app/src/main/jniLibs/arm64-v8a. It needs no desktop NDK, host toolchain, or PATH setup.

The plugin contributes a "COTG C++ Engine" item to the IDE sidebar and a "C++ Compiler" main editor tab. The tab hosts a build console with a Compile action, a Copy Logs button, a Download NDK shortcut, and two build options: Export libc++_shared.so and Parallel mode.

The implementation is two Kotlin files: cotgndk.kt (the plugin entry point and UI contributions) and cotgndkFragment.kt (the console UI and the compiler engine). The plugin requires the filesystem.read, filesystem.write, project.structure, system.commands, and ide.environment.write permissions, and consumes the IdeEnvironmentService, IdeProjectService, and IdeEditorTabService host services.

2. Core Functionality

The C++ Compiler console

The cotgndkFragment renders a terminal-style console (a TextView inside a ScrollView) and the build controls. When the tab opens, the fragment wires up its buttons and runs an NDK status check that decides whether compilation is available:

  • Compile: runs the build pipeline on a background coroutine
  • Copy Logs: copies the console text to the system clipboard
  • Download NDK: opens the NDK release URL in a browser
  • Export libc++_shared.so: copies the STL runtime into jniLibs when enabled
  • Parallel mode: selects make -j4 (fast) instead of make -j1 (safe)

NDK detection

On open, checkNdkStatus() looks for an NDK in two places and enables or disables the build controls based on what it finds:

ConditionResult
$ANDROID_HOME/cotgx_ultimate_ndk exists and is non-emptyNDK installed, ready to compile
cotgx-ndk-ultimate.tar.gz present in the Download folderArchive detected, extracted on first compile
Neither presentCompile disabled; Download NDK shown with instructions

NDK provisioning

On the first compile, if the NDK is not installed, the plugin copies the downloaded archive into the workspace and extracts it with the native tar command into the IDE's Android home directory. Extraction output streams to the console. 

val extractCmd = "tar -xzf '${tempFile.absolutePath}' -C '${targetDir.absolutePath}'"
val extProcess = ProcessBuilder("sh", "-c", extractCmd)
    .redirectErrorStream(true).start()

Toolchain discovery

The engine does not assume a fixed NDK layout. An eight-step shell script locates each required component at build time and stops with an error if a critical one is missing:

  • clang / clang++: the C and C++ compilers, made executable with chmod +x
  • sysroot: the Android system root for headers and libraries
  • jni.h: JNI headers, added with -isystem
  • c++/v1: the libc++ STL include root
  • libc++.so / libc++.a: the STL link target
  • bundled cmake and make, falling back to the IDE's copies if absent

The build targets aarch64-linux-android21 with --sysroot set to the discovered system root.

Sysroot self-healing

Some on-device sysroots ship without the Android platform stub libraries that native projects link against. Before configuring CMake, the engine back-fills any missing libraries from the device's /system/lib64 (or /system/lib) into the sysroot:

liblog libandroid libEGL libGLESv2 libGLESv3 libjnigraphics libmediandk libOpenSLES libvulkan libaaudio libcamera2ndk libnativewindow
liblog guard. If liblog is not in the sysroot and cannot be provisioned from the system, the console prints a warning that any find_library(... log) or -llog dependency will fail. The cause is reported before the build reaches the linker.

Configure, build, export

With the toolchain resolved, the engine runs CMake with a manual toolchain configuration (no NDK toolchain file required), then runs make: 

"$CMAKE_BIN" "${cppDir}" \
    -DCMAKE_SYSTEM_NAME=Android \
    -DCMAKE_SYSTEM_VERSION=21 \
    -DCMAKE_SYSROOT="$SYSROOT_DIR" \
    -DCMAKE_C_COMPILER="$CLANG_BIN" \
    -DCMAKE_CXX_COMPILER="$CLANGXX_BIN" \
    -DCMAKE_MAKE_PROGRAM="$MAKE_BIN" \
    -DCMAKE_C_FLAGS="$C_FLAGS" \
    -DCMAKE_CXX_FLAGS="$CXX_FLAGS" \
    -DCMAKE_BUILD_TYPE=Release \
    -G "Unix Makefiles"

"$MAKE_BIN" SHELL=/system/bin/sh -j4   # or -j1 in safe mode

Built *.so files are copied into app/src/main/jniLibs/arm64-v8a/. When Export libc++_shared.so is enabled, the STL runtime is located in the NDK and copied alongside them. The console ends with [OK] BUILD SUCCESS or a failure and its exit code.

4. Integration Points

cotg ndk integrates with the host IDE through three services and native command execution. It does not modify any main-app source. All work happens at runtime through the plugin framework and the device's own shell, CMake, and Clang.

4.1 Plugin lifecycle (IPlugin)

The manifest declares the entry point:

<meta-data android:name="plugin.main_class"
           android:value="com.cotg.cotgndk.cotgndk" />

initialize() stores the PluginContext in a global bridge so the fragment can reach host services. The plugin also implements UIExtension for the sidebar and EditorTabExtension for the editor tab.

4.2 Environment access (IdeEnvironmentService)

getAndroidHomeDirectory() locates the IDE's Android home. It is the install target for the NDK (cotgx_ultimate_ndk) and the base for sysroot discovery and provisioning. Requires ide.environment.write.

4.3 Project access (IdeProjectService)

getCurrentProject().rootDir resolves the active project so the engine can find app/src/main/cpp/CMakeLists.txt and write build artifacts back into jniLibs. Requires project.structure.

4.4 Editor tabs (IdeEditorTabService)

Tapping the sidebar item calls selectPluginTab("com_cotg_cotgndk_main") after isTabSystemAvailable() to focus or open the "C++ Compiler" tab.

4.5 Native command execution

The compiler engine shells out through ProcessBuilder("sh", "-c", ...) to run tar, cmake, make, and the discovered Clang binaries, streaming combined stdout and stderr to the console. This requires the system.commands permission. The fragment is themed through PluginFragmentHelper.getPluginInflater(PLUGIN_ID, inflater).

5. Deployment & Usage

Building

# debug
./gradlew assemblePluginDebug

# release
./gradlew assemblePlugin

This produces a .cgp package under build/plugin/ (cotgndk-debug.cgp or cotgndk.cgp). The plugin links the Code on the Go API as compileOnly from the repository's shared ../libs/plugin-api.jar.

Installation

  1. Open Code on the Go's plugin manager.
  2. Import the .cgp file.
  3. The IDE discovers com.cotg.cotgndk.cotgndk from the manifest metadata and loads it.
  4. Open COTG C++ Engine from the sidebar, or the C++ Compiler editor tab.

Preparing the NDK

  1. In the C++ Compiler tab, tap Download NDK or open the release URL directly.
  2. Place cotgx-ndk-ultimate.tar.gz in the device Download folder.
  3. Reopen the tab. The status check detects the archive and enables Compile.
  4. The archive is extracted to $ANDROID_HOME/cotgx_ultimate_ndk on the first build and reused afterward.

Compiling a project

  1. Open a project that contains app/src/main/cpp/CMakeLists.txt.
  2. Open the C++ Compiler tab.
  3. Optionally enable Export libc++_shared.so and Parallel mode.
  4. Tap Compile and read the streamed build log.
  5. On success, the .so files are written to app/src/main/jniLibs/arm64-v8a/.

6. Key Benefits

  • On-device native builds. Compile C++ and JNI for Android inside Code on the Go, without a desktop, remote build server, or cross-compilation setup.
  • Zero-configuration toolchain. The engine discovers Clang, the sysroot, JNI headers, the STL, and the bundled CMake and make at build time, so it adapts to the NDK layout instead of hard-coding paths.
  • Self-healing sysroot. Missing Android platform libraries are provisioned from the device system, and an unresolved liblog is reported before the linker stage.
  • Fast or safe. A toggle switches between parallel (make -j4) and single-threaded (make -j1) builds.
  • Transparent feedback. Each step streams to a live console with copy-to-clipboard.
  • Reusable install. The NDK is extracted once into the Android home and reused across builds; later compiles skip extraction.
