Dependency Analysis

Audit your project's Gradle dependencies from inside Code on the Go, and fix the easy wins in one tap. This page explains what the plugin reports, how it runs, and what the one-tap fix actually does.

What it does. Runs the Dependency Analysis Gradle Plugin (DAGP) against the open project and surfaces three kinds of advice: unused, undeclared transitive, and misconfigured dependencies. A one-tap Fix dependencies applies DAGP's recommended edits and re-analyzes.

1. The three kinds of advice

  • Unused — declared in a build.gradle but not referenced by your code. Safe to remove; shrinks your build graph.
  • Undeclared transitive — used directly in your code but only present because some other dependency drags it in. You should declare it explicitly so an upstream change can't silently break you.
  • Misconfigured — declared on the wrong configuration, e.g. api where implementation would do. Tightening this speeds up incremental builds.

2. How analysis runs

Why you care: the plugin doesn't edit your build files to analyze. It runs Gradle's :buildHealth task with a generated init script that applies DAGP to your project on the fly, so analysis is non-invasive. The advice is read back from build/reports/dependency-analysis/build-health-report.json.

DAGP and its libraries are bundled with the plugin, so analysis works offline — no network fetch the first time you run it.

3. The one-tap fix

Why you care: applying advice by hand across many modules is tedious. Fix dependencies delegates to DAGP's :fixDependencies task, which rewrites your build.gradle(.kts) files in place — removing unused declarations, adding undeclared ones, and moving misconfigured ones.

Before it writes. You confirm first ("Apply N fixes? edits app/build.gradle"). After the rewrite the plugin re-runs the analysis and shows you a summary: how many dependencies were removed, added, and reconfigured.

Note: rewriting Kotlin DSL build files is reliable; Groovy DSL rewriting is best-effort and may skip files with complex logic. If a build.gradle is open in the editor when the fix runs, reload it to pick up the on-disk edits.

4. Permissions

This plugin requests exactly two permissions: system.commands (to run the two Gradle tasks) and filesystem.read (to read the report). It does not request network access — everything it needs is bundled — and it does not write files itself; the build-file edits are performed by Gradle.
