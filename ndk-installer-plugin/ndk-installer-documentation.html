1. Executive Overview

NDK Installer is a Code on the Go plugin that automatically installs a bundled Android NDK (Native Development Kit) and CMake into the IDE's Android SDK directory when activated, and cleanly removes them when deactivated. Alongside the binary installation, it registers an "NDK Activity" project template that scaffolds a complete Android NDK project with C++ support, JNI bindings, and CMake build configuration.

The plugin ships a pre-packaged ndk-cmake.tar.xz archive as a bundled asset. On activation, a background coroutine extracts this archive into the IDE's Android home directory while logging progress. On deactivation, the plugin removes the ndk/ and cmake/ directories, ensuring a clean uninstall.

The plugin consists of a single Kotlin source file (177 lines), requires ide.environment.write, filesystem.read, and filesystem.write permissions, and demonstrates several plugin capabilities: asset extraction via IdeArchiveService, environment directory access via IdeEnvironmentService, file management via IdeFileService, and project template registration via IdeTemplateService.

2. Core Functionality

NDK and CMake installation

The plugin bundles an ndk-cmake.tar.xz archive containing pre-built NDK and CMake binaries. On activation, a coroutine launched on Dispatchers.IO extracts this archive into the IDE's Android home directory using IdeArchiveService.extract(). The extraction reports progress in MiB via a callback: 

val result = source.use { stream ->
    archive.extract(
        source = stream,
        format = ArchiveFormat.TAR_XZ,
        destination = targetDir
    ) { bytes, entry ->
        val mib = bytes / MIB
        val suffix = entry?.let { " ($it)" } ?: ""
        context.logger.debug("extract progress: ${mib} MiB$suffix")
    }
}

The result is checked for success or failure, with detailed logging in either case (file count and total bytes on success, error message and exception on failure).

Clean removal

When the plugin is deactivated, it removes both the ndk/ and cmake/ subdirectories from the Android home directory using IdeFileService.delete(). This ensures that disabling the plugin does not leave behind orphaned binaries.

NDK Activity project template

The plugin registers an "NDK Activity" template via IdeTemplateService that scaffolds a complete Android project with native C++ support. The template includes:

  • A Gradle wrapper and project-level build files
  • An app module with NDK, CMake, and JNI configuration
  • A native-lib.cpp with a sample JNI function
  • A CMakeLists.txt for the native build
  • Both Java and Kotlin MainActivity variants
  • Full Android resource set (layouts, drawables, launcher icons, themes)

JNI bridge

The generated native-lib.cpp contains a sample JNI function that returns a string from C++. The package name is transformed for JNI naming using a Pebble filter: 

extern "C" JNIEXPORT jstring JNICALL
Java_${{ PACKAGE_NAME | replace({".": "_"}) }}_MainActivity_stringFromJNI(
        JNIEnv* env,
        jobject /* this */) {
    return env->NewStringUTF("Hello from C++");
}

The corresponding CMake configuration links against the Android log library and builds the native code as a shared library (native-lib.so).

NDK installation

private fun installNdk() {
    val env = context.services.get(IdeEnvironmentService::class.java)
        ?: return fail("IdeEnvironmentService unavailable")
    val archive = context.services.get(IdeArchiveService::class.java)
        ?: return fail("IdeArchiveService unavailable")

    val targetDir = File(env.getAndroidHomeDirectory().absolutePath)

    val source = context.resources.openPluginAsset(NDK_ARCHIVE_NAME)
        ?: return fail("Bundled asset not found: $NDK_ARCHIVE_NAME")

    context.logger.info("Installing NDK to ${targetDir.absolutePath}")
    val result = source.use { stream ->
        archive.extract(
            source = stream,
            format = ArchiveFormat.TAR_XZ,
            destination = targetDir
        ) { bytes, entry ->
            val mib = bytes / MIB
            val suffix = entry?.let { " ($it)" } ?: ""
            context.logger.debug("extract progress: ${mib} MiB$suffix")
        }
    }

    when (result) {
        is ExtractResult.Success -> context.logger.info(
            "NDK install complete: ${result.filesExtracted} files, " +
            "${result.bytesWritten / MIB} MiB"
        )
        is ExtractResult.Failure -> context.logger.error(
            "NDK install failed: ${result.error.message}",
            result.error
        )
    }
}

NDK removal

private fun removeNdk() {
    val env = context.services.get(IdeEnvironmentService::class.java) ?: return
    val file = context.services.get(IdeFileService::class.java) ?: return
    val ndkDir = File(env.getAndroidHomeDirectory(), NDK_SUBDIR)
    if (ndkDir.exists()) {
        val deleted = file.delete(ndkDir)
        context.logger.info("ndk removal from ${ndkDir.absolutePath}: $deleted")
    }
    val cmakeDir = File(env.getAndroidHomeDirectory(), CMAKE_SUBDIR)
    if (cmakeDir.exists()) {
        val deleted = file.delete(cmakeDir)
        context.logger.info("cmake removal from ${cmakeDir.absolutePath}: $deleted")
    }
}

Asset download (build-time)

The ndk-cmake.tar.xz archive is not checked into version control. A custom Gradle task downloadAssets fetches it from the release server and verifies the MD5 checksum before accepting it: 

val downloadAssets by tasks.registering {
    doLast {
        val archiveUrl = "https://www.appdevforall.org/dev-assets/release/v8/ndk-cmake.tar.xz"
        val md5Url     = "https://www.appdevforall.org/dev-assets/release/v8/ndk-cmake.tar.xz.md5"

        val archiveBytes = httpDownload(archiveUrl)
        archiveFile.writeBytes(archiveBytes)

        val expected = httpDownload(md5Url).toString(Charsets.UTF_8).trim()
        val actual   = md5Of(archiveFile)

        if (!expected.equals(actual, ignoreCase = true))
            throw GradleException("MD5 checksum mismatch")
    }
}

4. Integration Points

NDK Installer interacts with the host IDE through four services. It does not modify any main-app source files — all integration happens at runtime through the plugin framework.

4.1 Plugin lifecycle (IPlugin)

NdkInstallerPlugin implements IPlugin. The manifest declares the entry point: 

<meta-data android:name="plugin.main_class"
           android:value="org.appdevforall.ndkinstaller.NdkInstallerPlugin" />

initialize() stores the PluginContext and obtains IdeTemplateService. activate() launches the background NDK install and registers the template. deactivate() cancels installation, removes binaries, and unregisters the template. dispose() cancels the coroutine scope.

4.2 Environment access (IdeEnvironmentService)

The plugin uses IdeEnvironmentService.getAndroidHomeDirectory() to locate the Android SDK root. This is the target directory for both NDK extraction and removal. The plugin must declare ide.environment.write permission to access this service.

4.3 Archive extraction (IdeArchiveService)

IdeArchiveService.extract() handles decompression and extraction of the TAR_XZ archive. The service accepts a source InputStream, a format enum, a destination directory, and an optional progress callback reporting bytes extracted and current entry name.

4.4 File management (IdeFileService)

IdeFileService.delete() recursively removes the ndk/ and cmake/ directories during deactivation. This provides a safe, IDE-managed file deletion.

4.5 Template registration (IdeTemplateService)

The plugin uses the template service's builder API to assemble and register a .cgt template package. The builder supports:

  • description() — sets the template description
  • showPackageNameOption(), showLanguageOption(), showMinSdkOption() — enable creation wizard fields
  • thumbnailFromAssets() — sets the template thumbnail
  • addStaticFromAssets() — registers template files from bundled assets
  • build() — produces the .cgt package

On deactivation, the template is unregistered via unregisterTemplate("NDK.cgt").

5. Deployment & Usage

Downloading assets

Before building, fetch the bundled NDK/CMake archive: 

./gradlew :ndk-installer-plugin:downloadAssets

This downloads ndk-cmake.tar.xz from the release server, verifies the MD5 checksum, and places it in src/main/assets/.

Building

./gradlew :ndk-installer-plugin:assemblePluginDebug

This produces a .cgp plugin package under ndk-installer-plugin/build/outputs/. The build configuration sets noCompress += listOf("xz") to prevent the Android asset system from re-compressing the archive.

Installation

  1. Open Code on the Go's plugin manager.
  2. Import the .cgp file.
  3. The IDE discovers NdkInstallerPlugin via the manifest metadata and activates it automatically.
  4. The NDK and CMake binaries are extracted in the background — progress is logged to the plugin logger.

Creating an NDK project

  1. Open the new-project wizard in Code on the Go.
  2. Select the "NDK Activity" template.
  3. Enter the app name, package name, and save location.
  4. Optionally choose the language (Java or Kotlin) and minimum SDK.
  5. The wizard generates a complete project with C++ source, JNI bindings, CMake configuration, and a ready-to-build Gradle setup.

Uninstallation

Deactivating or removing the plugin automatically deletes the ndk/ and cmake/ directories from the Android home and unregisters the NDK Activity template. No manual cleanup is required.

6. Key Benefits

  • Zero-configuration NDK setup. Developers get a working NDK and CMake installation simply by enabling the plugin — no manual downloads, no PATH configuration, no version hunting.
  • Clean install/uninstall. The plugin fully manages the NDK lifecycle: installation on activation, removal on deactivation. There are no orphaned binaries or lingering directories after the plugin is disabled.
  • Ready-to-build project template. The NDK Activity template scaffolds a complete project with JNI, CMake, Gradle, and proper ABI configuration. Developers can build and run native C++ code immediately after project creation.
  • Background installation. NDK extraction runs on a background coroutine (Dispatchers.IO), so the IDE remains responsive during the multi-megabyte extraction process.
  • Integrity verification. The build-time downloadAssets task verifies the archive's MD5 checksum, ensuring the bundled NDK has not been corrupted or tampered with.
  • Compact implementation. The entire plugin is a single 177-line Kotlin file, making it a clear reference for building plugins that bundle and install large binary assets, interact with the IDE's environment and archive services, and register project templates.
