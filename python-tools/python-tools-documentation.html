1. Executive Overview

Python Tools is a Code on the Go plugin that adds two new project types, a Flask web app and a minimal Python starter, and makes them runnable on the device. It is a reference example of how a plugin adds a language and its project templates.

When the open project belongs to the Python domain, the plugin hides the built-in Gradle build actions and contributes Python-native ones in their place: Run app, Run current file, Install requirements, and Run tests. Projects that are not Python (anything carrying a Gradle build file) are left untouched.

On activation the plugin also ensures a Python interpreter is available. If python is not on the path, it installs one, so a fresh device can create and run a Python project with no manual setup.

Only light testing has been done; use at your own risk. Customer support cannot provide help with this plugin.

2. Core Functionality

Project templates

The plugin registers two templates with the host's template service. Both render from .peb (Pebble) assets bundled in the .cgp:

  • Python Flask App: a Flask web app with routes, HTML templates, static files, a configurable port, and basic error handling (app.py, config.py, templates/, static/, requirements.txt).
  • Python Starter: a minimal project with a single main.py entry point and a requirements.txt.

Python-native build actions

While a Python project is open, the plugin contributes the following actions. The most-used action leads, so it takes the prime slot:

ActionRunsWhen
Run appThe project entry point (app.py → main.py → manage.py → __main__.py → wsgi.py)Always
Run current filepython <focused file>When a .py file is focused
Install requirementspip install -r requirements.txtAlways
Run testspytest (installed on demand)Always

Quality-of-life behaviour

  • Live output. Commands run with PYTHONUNBUFFERED=1, so app output and pip progress stream into the build panel instead of appearing only at exit.
  • Auto-install on a missing import. If a run fails with ModuleNotFoundError or ImportError, the plugin notifies the user and runs pip install -r requirements.txt.
  • Clean cancel. Cancelling a run stops the whole process tree, including a Flask reloader child, so the server releases its port.

3. Technical Architecture

The plugin is two Kotlin files plus its assets. It implements IPlugin for lifecycle and BuildActionExtension for the build actions, and reaches the host through services it looks up at initialize().

  • python-tools/
    • src/main/kotlin/…/plugin/
      • PythonToolsPlugin.kt (templates, build actions, interpreter bootstrap)
      • PythonDomain.kt (Python-project detection)
    • src/main/assets/templates/ (flask/ and starter/ .peb sources)
    • src/main/res/drawable/ (build-action icons)

Domain gating

PythonDomain decides whether a directory is a Python project. A Gradle marker disqualifies it, so a stray script inside a Java or Kotlin project never steals the Gradle build actions: 

private fun detect(root: File): Boolean {
    if (GRADLE_MARKERS.any { File(root, it).exists() }) return false
    if (File(root, "requirements.txt").exists()) return true
    if (File(root, "pyproject.toml").exists()) return true
    return root.listFiles { f -> f.isFile && f.name.endsWith(".py") }
        ?.isNotEmpty() == true
}

Both toolbarActionsToHide() and getBuildActions() gate on this check, returning an empty set or list for non-Python projects.

Hiding the Gradle actions

The host allows only a fixed set of build actions to be hidden. The plugin requests that set when a Python project is open: 

override fun toolbarActionsToHide(): Set<String> =
    if (isPythonProjectOpen()) ToolbarActionIds.BUILD_HIDEABLE else emptySet()

Process lifecycle

Commands are shell commands (sh -c) run through IdeCommandService. The entry point is launched with exec, so the Python process, not a wrapper shell, is the one the host tracks. To make cancel tear down a Flask reloader's worker child, the plugin ships a small sitecustomize.py on PYTHONPATH that sets the kernel parent-death signal: 

# sitecustomize.py (installed into the plugin dir, added to PYTHONPATH)
import ctypes, signal
ctypes.CDLL("libc.so").prctl(1, signal.SIGKILL)  # PR_SET_PDEATHSIG

When the host kills the tracked process, the kernel kills its children too, freeing the server's port.

4. Integration Points

Every host touch-point is an interface from plugin-api.jar. The plugin declares the permissions filesystem.read, filesystem.write, system.commands, and project.structure.

Extension / ServiceUsed for
IPluginLifecycle: initialize, activate, deactivate, dispose.
BuildActionExtensionContribute build actions and hide the Gradle ones; receive start and complete callbacks.
IdeTemplateServiceBuild and register the Flask and Starter templates from assets.
IdeProjectServiceResolve the open project's root for domain detection.
IdeEditorServiceRead the focused file for Run current file.
IdeCommandServiceRun shell commands (run, pip, pytest, interpreter install).
IdeUIServiceSurface notices (install prompts, run failures, stop notices).

5. Deployment & Usage

Build the installable plugin from the plugin folder:

cd python-tools
./gradlew assemblePlugin

The resulting .cgp lands under build/plugin/. Install it from inside Code on the Go via the Plugin Manager. Then:

  1. Create a project from Python Flask App or Python Starter in the new-project picker.
  2. Open it. The toolbar swaps to the Python build actions.
  3. Tap Run app. On first run the plugin installs Python if needed, then starts the app; output streams into the build panel.

6. Key Benefits

  • Adds a language end to end. Covers the full path from project templates, through a domain check, to a language-native set of build actions.
  • Domain-safe gating. A reusable pattern for acting only on the plugin's own project type and leaving other projects untouched.
  • On-device interpreter setup. Installs the runtime when it is missing, so the feature works on a fresh device.
  • Reliable process lifecycle. A self-contained technique for making cancel stop a server that forks a child.
