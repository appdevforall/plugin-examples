1. Executive Overview

Random XKCD is a Code on the Go plugin that puts an xkcd-comic browser into the editor's bottom sheet. A new XKCD tab joins the built-in bottom-sheet tabs (Build Output, App Logs, …); inside it, a five-button navigation row mirrors the xkcd.com control bar — |< · < Prev · Random · Next > · >| — with the comic image rendered below. The panel opens on the latest comic and stays there until the user navigates.

The image itself responds to two standard gestures: a single tap copies the comic's canonical URL to the clipboard, and a double tap copies the PNG itself (routed through the host IDE's FileProvider, since plugin manifests don't register providers of their own). Long-pressing the bottom-sheet tab opens a three-tier tooltip whose Tier 3 button launches the full code walkthrough that ships inside the .cgp.

The plugin spans three Kotlin source files and one XML layout (~600 lines of code total), declares only the network.access permission, and is deliberately written as a canonical “this is what a small CoGo plugin looks like” example. Every extension point it touches — IPlugin, UIExtension, DocumentationExtension — is exercised once, in the simplest way that still tells the truth.

2. Core Functionality

Bottom-sheet panel

The plugin contributes a single XKCD tab to the editor bottom sheet, ordered at 200 among plugin tabs. The tab body is a fragment with three regions, stacked vertically:

  • A five-button navigation row at the top.
  • The comic image inside a FrameLayout card.
  • The comic caption (number + title) and the xkcd alt-text beneath it.

A progress spinner and an offline empty-state share the same region as the image and swap in as needed.

Navigation row

The button row mirrors xkcd.com's own nav. Each button calls loadComic() with a Navigation enum value; the fragment debounces concurrent fetches and re-renders on the main thread.

ButtonActionDisabled when
|< FirstFetch comic #1never
< PrevFetch current − 1at comic #1 or before first load
RandomPick a random number in [1, latest]never
Next >Fetch current + 1at the high-water mark or before first load
>| LastRe-probe the latest comicnever

Image gestures

A standard GestureDetector.SimpleOnGestureListener attached to the ImageView (not the whole panel, so scrolling a tall comic doesn't trigger spurious taps) handles two gestures:

GestureEffect
Single tapCopy https://xkcd.com/<num>/ to clipboard as plain text
Double tapCopy the in-memory PNG bytes to clipboard as image/png

On Android 13+ the system shows its own clipboard indicator, so the plugin suppresses its toast on those versions to avoid double-up.

Network surface

All HTTP lives in XkcdApiClient, in one file so it reads top-to-bottom. Two JSON endpoints, no auth, no rate-limit handling required:

EndpointPurpose
GET https://xkcd.com/info.0.jsonLatest comic metadata
GET https://xkcd.com/<num>/info.0.jsonSpecific comic metadata
GET https://imgs.xkcd.com/…Comic PNG (streamed, capped at 5 MB)

Defensive parsing

The JSON parser rejects any img URL that doesn't start with https://imgs.xkcd.com/, so a malicious or compromised metadata response can't point the bitmap decoder at an attacker-controlled host. JSON responses are bounded by MAX_JSON_BYTES (64 KB) and image downloads by MAX_IMAGE_BYTES (5 MB), both well above what xkcd actually returns. Every IO failure converges on a single null return that routes through the offline empty-state.

The #404 quirk

xkcd comic #404 is a joke — its JSON endpoint returns HTTP 404 deliberately. Both the random picker and the Prev/Next walker skip past it: Random retries until it lands somewhere else, and Prev/Next continue stepping in the same direction (so Prev from #405 lands on #403, Next from #403 lands on #405). The user never sees a “failed to load” toast for this case.

Three-tier tooltip

Long-pressing the XKCD tab opens the three-tier help surface contributed via DocumentationExtension:

TierSourceTrigger
1 — one-linerPluginTooltipEntry.summaryLong-press the tab
2 — HTML paragraphPluginTooltipEntry.detail“See More” on the Tier 1 popup
3 — full HTML pagesrc/main/assets/docs/index.html“Code walkthrough” button inside Tier 2

The Tier 3 page is served by the host IDE at http://localhost:6174/plugin/org.appdevforall.randomxkcd/index.html and indexed at install time by the host's Tier3AssetWalker.

Data flow

user taps a nav button (|< · < Prev · Random · Next > · >|) | v XkcdPanelFragment.loadComic(Navigation) | | Dispatchers.IO v XkcdApiClient ├─ fetchLatest() ──> https://xkcd.com/info.0.json ├─ fetchByNumber(n) ──> https://xkcd.com/n/info.0.json ├─ fetchRandom() ──> loops fetchByNumber, skipping #404 └─ openImageStream() ──> https://imgs.xkcd.com/... | | parseComic() — rejects non-imgs.xkcd.com hosts v XkcdComic + raw PNG bytes | | main thread v showComic() — renders bitmap, caption, alt | | tap / double-tap v ClipboardManager (text URL OR image via host FileProvider)

Lifecycle and navigation state

XkcdPanelFragment tracks three pieces of state across button presses:

FieldPurpose
currentComicNumNumber currently displayed. Drives Prev / Next enable state.
latestComicNumHigh-water mark from the most recent Latest probe (or any Random that landed higher). Caps Next.
lastBytesRaw PNG of the displayed comic, kept so “copy image” doesn't re-download.

A single loadJob guards against fan-out from rapid button presses — if a fetch is already in flight, subsequent taps are dropped on the floor.

Cooperative cancellation

Both fetchRandom's retry loop and the image-download read loop call coroutineContext.ensureActive() on every iteration, so when the host tears the fragment down (rotation, tab close) the coroutine exits cleanly with a CancellationException rather than running to completion against a dead view.

4. Integration Points

Random XKCD touches the host IDE through three extension interfaces. It does not consume any IDE services beyond the host's FileProvider authority — all behavior is self-contained inside the plugin's own sandbox.

4.1 Plugin lifecycle (IPlugin)

XkcdRandomPlugin implements IPlugin. The manifest declares the entry point: 

<meta-data android:name="plugin.main_class"
           android:value="org.appdevforall.randomxkcd.XkcdRandomPlugin" />

initialize() stores the PluginContext and is wrapped in try/catch so a stray exception in setup can't crash the host. activate(), deactivate(), and dispose() log lifecycle transitions; there's no per-project state to clean up.

4.2 Bottom-sheet tab (UIExtension)

getEditorTabs() contributes a single tab to the editor bottom sheet: 

TabItem(
    id = TAB_ID,                              // "xkcd_bottom_tab"
    title = "XKCD",
    fragmentFactory = { XkcdPanelFragment() },
    order = 200,
    tooltipTag = TOOLTIP_TAG_TAB              // "xkcd.tab"
)

The fragmentFactory returns a new fragment on every invocation — never a singleton, because the host manages fragment lifecycles. The tooltipTag wires the tab to the corresponding PluginTooltipEntry below.

4.3 Three-tier docs (DocumentationExtension)

Two methods feed the tooltip system. getTooltipEntries() returns a single PluginTooltipEntry with summary, detail HTML, and a Code walkthrough button whose URI resolves to index.html inside the Tier 3 asset directory. getTier3DocsAssetPath() returns "docs", telling the host that everything under src/main/assets/docs/ should be indexed and served from http://localhost:6174/plugin/org.appdevforall.randomxkcd/….

4.4 Clipboard image — the FileProvider hop

Copying an image to the system clipboard requires a content URI. Plugins can't register their own FileProvider at runtime (they're loaded via DexClassLoader, not installed as apps), so the double-tap path routes through the host's existing FileProvider authority:

  1. Write the in-memory PNG to filesDir/xkcd_share/last.png on Dispatchers.IO.
  2. Ask the host's FileProvider for a content URI: FileProvider.getUriForFile(ctx, "${ctx.packageName}.providers.fileprovider", target).
  3. Wrap the URI in a ClipData.newUri clip — this queries the resolver for the URI's MIME type, advertising image/* to paste targets.

The host's file_provider_paths.xml exposes filesDir and the provider declares grantUriPermissions="true", so the system can hand out a temporary read grant to whatever app calls ContentResolver.openInputStream on the clip URI.

Known caveat: some OEM clipboard managers have been observed to silently drop image clips delivered this way. The flow works on stock Android API 24+; real-device verification is recommended for OEM coverage.

4.5 Plugin-scoped LayoutInflater

XkcdPanelFragment.onGetLayoutInflater wraps the inflater via PluginFragmentHelper.getPluginInflater. Without this, R.layout.fragment_xkcd_panel resolves against the host IDE's resources rather than the plugin's APK and throws Resources$NotFoundException at inflate time.

4.6 Permissions

The manifest declares only network.access: 

<meta-data android:name="plugin.permissions"
           android:value="network.access" />

Writes to the plugin's own filesDir (the FileProvider hop) and access to the system clipboard are sandbox-allowed and need no declaration. CoGo's filesystem.* permissions only gate the host IDE's project-file services — not used here.

5. Deployment & Usage

Building

cd random-xkcd
./gradlew assemblePlugin

Produces random-xkcd/build/plugin/random-xkcd.cgp — the bundle you sideload into Code on the Go.

Installation

  1. Open Preferences → Plugin Manager → +.
  2. Select the random-xkcd.cgp file.
  3. The IDE discovers XkcdRandomPlugin via the manifest metadata and activates it automatically.
  4. Open any project — the XKCD tab appears in the editor bottom sheet.

Using the panel

  1. Tap the XKCD tab in the bottom sheet — the latest comic loads automatically.
  2. Navigate with the five-button row above the comic.
  3. Single-tap the image to copy the comic's URL; double-tap to copy the image itself.
  4. Long-press the XKCD tab for the three-tier help surface, including the full code walkthrough.

Reading the code walkthrough

  • Inside CoGo (the canonical path) — long-press the XKCD tab → tap See More → tap Code walkthrough. The IDE opens the page in an in-IDE WebView.
  • Outside CoGo — open src/main/assets/docs/index.html directly in any browser. Renders identically.

6. Key Benefits

  • A canonical small-plugin example. Under 600 lines of Kotlin across three source files, every plugin-specific concept called out exactly where it shows up — the in-asset walkthrough explains each one as a numbered step.
  • Real-world UI patterns. Demonstrates a bottom-sheet tab, fragment-based UI, a plugin-scoped LayoutInflater, and the standard Android GestureDetector idiom for single/double-tap distinction.
  • Real-world network patterns. Bounded reads, host whitelisting on parsed URLs, cooperative cancellation under lifecycleScope, and a single fetch-in-flight guard that survives rapid button mashes.
  • Cross-app image clipboard. Shows the practical FileProvider hop a plugin needs in order to put image/* data on the system clipboard — including the OEM-clipboard caveat worth knowing before relying on it.
  • Three-tier in-IDE help. Wires up DocumentationExtension end-to-end — one-liner, HTML paragraph, and a full asset-served walkthrough — serving as a template for any plugin that wants discoverable, in-context documentation.
  • Sandbox-honest permissions. Declares only network.access and documents why the clipboard and plugin-private filesystem writes do not need declarations — useful reference for understanding CoGo's permission surface.
