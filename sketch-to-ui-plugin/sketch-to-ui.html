1. Executive Overview

Sketch to UI is a Code on the Go plugin that turns a picture of an interface — a whiteboard sketch, a paper drawing, or a screenshot of another app — into a working Android XML layout. The user opens a layout XML file, picks an image, and the plugin runs a complete computer-vision pipeline (object detection + OCR + layout assembly) that writes the result back into the editor.

All inference runs on-device. A bundled YOLO model (best_float32.tflite, ~24 MB) detects widget shapes; Google ML Kit text recognition extracts labels; a margin-annotation parser reads handwritten attribute hints (like T-1 / B-1) from the left and right sides of the sketch; and a layout tree builder emits a LinearLayout-based XML structure scaled to a 360×640 dp target. The plugin also updates res/values/strings.xml in the host project with the strings it extracted, so the generated layout is fully wired up to resources.

The plugin spans ~84 source files across UI, domain, data, and DI layers, requires filesystem.read, filesystem.write, project.structure, and native.code permissions (the last is for the TFLite GPU delegate), and demonstrates a fully self-contained ML workflow embedded in a CoGo plugin.

2. Core Functionality

Activation

The plugin contributes a single main-menu item, "Sketch to UI", with the eye-shaped icon. The item is enabled only when the file currently open in the editor is an Android layout resource (an .xml file whose parent directory is named layout). When the user activates it, the plugin remembers the target layout file's path and opens its own full-screen workspace inside the IDE.

The workspace screen

The plugin's screen is a single SketchToUiFragment that hosts:

  • A toolbar with the title Generate XML and a back button.
  • A large ZoomableImageView for the source picture. Tapping an empty view opens the system file picker; tapping detected image placeholders opens a secondary picker so the user can bind a real drawable to that region.
  • A GuidelinesView overlay with two draggable vertical guides. Everything between them is treated as the layout canvas; everything outside is treated as margin annotation space.
  • Three action buttons: Detect, Update, and Save.

Detected widget types

The bundled YOLO model recognises 13 labels:

dropdown_symbolThe chevron of a Spinner.
image_placeholderEmpty image slot; bindable to a drawable in the UI.

switch_on, switch_offMaterial switch in either state.

Margin annotations

The plugin lets the sketch carry instructions that wouldn't fit on the drawing itself. A tag like T-1 drawn beside a widget binds it to a free-text note like T-1: hint="Email" written in the left or right margin. MarginAnnotationParser partitions detections into three zones (left margin, canvas, right margin), WidgetAnnotationMatcher joins each canvas tag to its margin note, and FuzzyAttributeParser resolves the note (using fuzzy matching) into concrete Android XML attributes.

Three terminal actions

ButtonEffect
Detect Runs the full vision pipeline on the loaded image, overlays colour-coded boxes on the picture, and stages a generated layout internally.
Update Asks for confirmation, then replaces the contents of the target layout file (the one the user had open when launching the plugin) with the generated XML. Also merges any extracted strings into app/src/main/res/values/strings.xml. Returns the user to the editor.
Save Writes the generated layout XML to the device's Downloads/ folder under a generated filename, leaving the project untouched.

3. The Vision Pipeline

Pressing Detect runs RunVisionUC, which orchestrates a four-stage pipeline. Each stage reports progress back to the UI so the screen-reader can announce it.

Source bitmap | v +----------------------------+ | 1. YOLO detection | RunningYolo | best_float32.tflite | | GPU-delegated via | | LiteRT | +-------------+--------------+ | v raw bounding boxes & labels +----------------------------+ | 2. Region OCR | RunningOcr | ML Kit text-recognition | | per detected region | | + full-image pass for | | untagged text | +-------------+--------------+ | v enriched + remaining + margin detections +----------------------------+ | 3. Detection merging | MergingDetections | DetectionMerger, | | MarginAnnotationParser | +-------------+--------------+ | v canvas detections + annotation map +----------------------------+ | 4. Layout assembly | GeneratingXml | YoloToXmlConverter | | LayoutTreeBuilder | | AndroidXmlGenerator | +-------------+--------------+ | v layout.xml + strings.xml fragment

Stage 1 — YOLO detection

VisionRepositoryImpl loads the bundled best_float32.tflite from assets and runs inference through LiteRT, using the GPU delegate when available. Raw output is post-processed into DetectionResult records. GenericBoxResolver then promotes generic boxes into specific widgets when neighbouring context (text content, tag proximity) makes the type obvious.

Stage 2 — Region OCR

RegionOcrProcessor runs ML Kit text recognition on each bounding box plus a full-image pass. Per-region OCR gives each widget its label (button text, checkbox caption, etc.); the full-image pass picks up any text that didn't fall inside a detection so it can be turned into a standalone TextView.

Stage 3 — Merging & margin parsing

DetectionMerger combines enriched widgets, remaining text blocks, and image-wide OCR. Detections whose centre falls outside the user-drawn guidelines are routed to MarginAnnotationParser, which extracts free-text notes from each gutter and produces a Map<widgetTag, annotation>.

Stage 4 — Layout assembly

YoloToXmlConverter takes the canvas detections, scales their pixel boxes into a 360×640 dp coordinate system with DetectionScaler, attaches loose text via TextAssociator, applies margin annotations through WidgetAnnotationMatcher + FuzzyAttributeParser, then hands the cleaned widget list to LayoutTreeBuilder and AndroidXmlGenerator to emit a LinearLayout-based XML tree wrapped in a ScrollView. String literals extracted along the way are returned separately so they can be merged into the project's strings.xml.

Coordinate system & scaling

Detections enter the pipeline in source-image pixels. The final output is always scaled to a logical 360×640 dp canvas (TARGET_DP_WIDTH × TARGET_DP_HEIGHT in SketchToUiFragment), so the produced layout looks reasonable regardless of the source image's dimensions. The horizontal guidelines default to 20 % and 80 % of the image width and can be dragged by the user before pressing Detect.

5. Integration Points

Sketch to UI talks to the host IDE through three extension interfaces and four services. It never modifies any IDE source — integration is entirely through the plugin framework.

5.1 Plugin lifecycle (IPlugin)

The manifest declares the entry point: 

<meta-data android:name="plugin.main_class"
           android:value="com.appdevforall.sketchtoui.plugin.SketchToUiPlugin" />

initialize() stores the PluginContext and caches a static Android Context reference (used later by the fragment to access plugin assets). activate() / deactivate() are no-ops; dispose() releases nothing on purpose — the native TFLite resources are released after each detection in RunVisionUC's finally block.

5.2 Main-menu entry (UIExtension)

getMainMenuItems() contributes a single item: 

MenuItem(
    id = "sketch_to_ui",
    title = "Sketch to UI",
    icon = R.drawable.ic_computer_vision,
    action = { openSketchToUiIfValid() },
    tooltipTag = TOOLTIP_SKETCH_TO_UI
).apply {
    isEnabledProvider = { hasCurrentLayoutXml() }
}

The isEnabledProvider is queried on every menu render so the item greys out as soon as the user switches away from a layout XML file. Side-menu items and toolbar actions are empty for this plugin.

5.3 Documentation tooltip (DocumentationExtension)

The plugin contributes one tooltip entry under category plugin_com.appdevforall.sketchtoui.plugin with a summary shown on hover and an HTML detail body shown in the help drawer.

5.4 Services consumed

ServiceUsage
IdeEditorService getCurrentFile() — check whether a layout XML is open before enabling the menu item, and read the editor's current file as a fallback target. getLineCount() / getLineText() / replaceRange() — rewrite the open buffer in place so the user sees the change without reloading.
IdeUIService openPluginScreen() — opens the workspace fragment as a full-screen plugin destination titled Generate XML.
IdeProjectService getCurrentProject().rootDir — locates the project root so the GeneratedStringsWriter can find app/src/main/res/values/strings.xml.
IdeFileService readFile() / writeFile() — reads the existing strings.xml, writes back the merged version, and falls back to writing the layout file directly if the editor buffer isn't the right target.

5.5 Fragment infrastructure

SketchToUiFragment uses PluginFragmentHelper.getPluginInflater() so resources bundled inside the .cgp resolve at inflation time, and PluginFragmentHelper.getServiceRegistry() to obtain IDE services from inside the fragment lifecycle.

5.6 Permissions

6. Deployment & Usage

Building

cd sketch-to-ui-plugin
./gradlew assemblePlugin

Produces build/plugin/sketch-to-ui.cgp. The libs/plugin-api.jar at the repo root must exist; refresh it with ../scripts/update-libs.sh if you have just synced the upstream CoGo source.

Installation

  1. Open Code on the Go's Plugin Manager.
  2. Import the .cgp file.
  3. CoGo discovers SketchToUiPlugin via the manifest metadata and activates it automatically. The "Sketch to UI" entry appears in the main menu.

Generating a layout

  1. Open the layout file you want to overwrite (e.g. app/src/main/res/layout/activity_main.xml).
  2. Activate the Sketch to UI menu item. The workspace opens.
  3. Tap the central image area and pick a photo or screenshot.
  4. Drag the two vertical guidelines so the canvas (the part of the image that represents the screen) is between them and any margin annotations are outside.
  5. Tap Detect. Status messages cycle through model load, YOLO, OCR, merge, and XML generation. Detected boxes are drawn over the image when the run finishes.
  6. Tap any green image placeholder box to bind a drawable to it. The picked image is imported into res/drawable and the layout regenerates with a reference to it. Tap the small delete icon on a placeholder to clear that binding and remove the imported file.
  7. Tap Update and confirm to overwrite the target layout file. The plugin also appends any extracted <string> entries to your project's strings.xml, then returns you to the editor.
  8. Alternatively, tap Save to write the layout XML to the device's Downloads folder without touching the project.

Margin annotation conventions

Margin notes are free-text but the fuzzy parser recognises Android attribute names. For example, writing T-1: hint Email Address in the right margin and drawing the T-1 tag next to an EditText on the canvas will produce android:hint="Email Address" on that widget. See the FuzzyAttributeParserTest in src/test/ for the supported attribute vocabulary.

8. Key Benefits

  • From idea to running screen in one step. A photo of a whiteboard becomes a real, editable layout XML file inside the project — no transcription pass.
  • Annotations on the sketch itself. Tag-and-margin notation lets the designer carry intent that a pure picture can't express, without forcing the developer to retype it.
  • Fully on-device. No network calls and no third-party service — the YOLO model and OCR engine both run locally, so the plugin works offline and keeps sketches private.
  • Round-trips into the project. Generated strings are merged into strings.xml and image placeholders can be bound to real drawables that get imported into res/drawable, so the result is wired up to the rest of the project, not a stand-alone snippet.
  • Comprehensive plugin-API example. Demonstrates main menu contribution, full-screen plugin destinations, documentation tooltips, multiple IDE services, fragment-based UI with PluginFragmentHelper, asset packaging, native library permissions, and on-device ML — making it a substantial reference for non-trivial plugins.
